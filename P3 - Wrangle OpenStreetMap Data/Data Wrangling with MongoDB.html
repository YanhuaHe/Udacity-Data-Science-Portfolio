"""check the "k" value for each "<tag>" and see if they can be valid keys in MongoDB, as well as see if there are any other 
   potential problems. we have a count of each of four tag categories in a dictionary:
  "lower", for tags that contain only lowercase letters and are valid,
  "lower_colon", for otherwise valid tags with a colon in their names,
  "problemchars", for tags with problematic characters, and
  "other", for other tags that do not fall into the other three categories.
"""
lower = re.compile(r'^([a-z]|_)*$')
lower_colon = re.compile(r'^([a-z]|_)*:([a-z]|_)*$')
problemchars = re.compile(r'[=\+/&<>;\'"\?%#$@\,\. \t\r\n]')

def key_type(element, keys):
    if element.tag == "tag":
        key_value = element.attrib["k"]
        if lower.search(key_value):
            keys["lower"] += 1
        elif lower_colon.search(key_value):
            keys["lower_colon"] += 1
        elif problemchars.search(key_value):
            keys["problemchars"] +=1
        else:
            keys["other"] +=1
    return keys

def process_map(filename):
    keys = {"lower": 0, "lower_colon": 0, "problemchars": 0, "other": 0}
    for _, element in ET.iterparse(filename):
        keys = key_type(element, keys)
    return keys

def audit_street_type(street_types, street_name):
    m = street_type_re.search(street_name)
    if m:
        street_type = m.group()
        if street_type not in expected:
            street_types[street_type].add(street_name)

def is_street_name(elem):

            part = mapping[part]

The following is my prefered data model to use.¶

"""{ "id": "2406124091", "type: "node", "visible":"true", "created": { "version":"2", "changeset":"17206049", "timestamp":"2013-08-03T16:43:42Z", "user":"linuxUser16", "uid":"1219059" }, "pos": [41.9757030, -87.6921867], "address": { "housenumber": "5157", "postcode": "60625", "street": "North Lincoln Ave" }, "amenity": "restaurant", "cuisine": "mexican", "name": "La Cabana De Don Luis", "phone": "1 (773)-271-5176"}"""

Based on the model, in particular the following things should be done:

  • process only 2 types of top level tags: "node" and "way"
  • all attributes of "node" and "way" should be turned into regular key/value pairs, except:
    • attributes in the CREATED array should be added under a key "created"
    • attributes for latitude and longitude should be added to a "pos" array, for use in geospacial indexing. Make sure the values inside "pos" array are floats and not strings.
  • if the second level tag "k" value contains problematic characters, it should be ignored
  • if the second level tag "k" value starts with "addr:", it should be added to a dictionary "address"
  • if the second level tag "k" value does not start with "addr:", but contains ":", you can process it in a way that you feel is best. For example, you might split it into a two-level dictionary like with "addr:", or otherwise convert the ":" to create a valid key.
  • if there is a second ":" that separates the type/direction of a street, the tag should be ignored.

def shape_element(element):
    node = {}
    if element.tag == "node" or element.tag == "way" :
        nd_info = []
        address_info = {}
        node["created"] = {}
        node["id"] = element.attrib["id"]
        node["type"] = element.tag
        if "visible" in element.attrib.keys():
            node["visible"] = element.attrib["visible"]
            
        # attributes for latitude and longitude should be added to a "pos" array,
        if "lat" in element.attrib.keys():
            node["pos"] = [float(element.attrib["lat"]),float(element.attrib["lon"])]
        
        for i in CREATED:
            node["created"][i]=element.attrib[i]   
               
        for tag in element.iter("tag"):
            
            # if the second level tag "k" value contains problematic characters, it should be ignored
            if problemchars.search(tag.attrib["k"]):
                continue 
                
            # check if there is "addr:" in the tag attributes' key
            elif tag.attrib["k"].startswith("addr:"): 
                # if there is a second ":" that separates the type/direction of a street, the tag should be ignored.
                if ":" in tag.attrib["k"][5:]:
                    continue 
                else:
                    # create a key using the string after "addr:"
                    key = tag.attrib["k"][5:]
  
                    # check if the key is "street", if so, update the street name
                    if key == "street":
                        value = update_name(tag.attrib["v"], mapping)
                        address_info[key] = value                    
                    else:
                        address_info[key] = tag.attrib["v"]
                    
                    if address_info:
                        node["address"] = address_info
            else:
                node[tag.attrib['k']] = tag.attrib['v']
        
        # for "way" specifically, turn the value of nd 'ref' key into a list   
        for tag2 in element.iter("nd"):
            nd_info.append(tag2.attrib['ref'])
        
        if nd_info != []:
            node['node_refs'] = nd_info
        
        # check if there is "postcode" in the element, if so, check whether it is within the scope of San Jose. If so, 
        # standardize the postcode. Otherwise, ignore this element. 
        if "address" in node:
            if "city" in node["address"]:
                if node["address"]["city"] not in ["San Jose", "san jose",'san Jose','San jose']:
                    return None 
            if "postcode" in node["address"]:
                value = node["address"]["postcode"]
                if update_postcode(value) is False:
                    return None  
                else:
                    value = update_postcode(value)
                    node["address"]["postcode"] = value
        return node

Conclusion¶

After the review of the data, I think the overall quality and quantity of San Jose OpenStreetMap data are good. A few improvement areas are:

1.Address format standardization: publish a gold standard on OpenStreetMap website for users and enter the address according to the standards, system-supported auto-correcting tool to help users adjust their input according to gold standard. Encourage users to incorporate more complete information such as zipcode, type of amenities or so on. 2.Autofilter for mistaken information, in the scope of city San Jose, it is obvious an error to input location in other nearby cities with a city name as "Sunnyvale", "Saratoga" or so on. I think it is better to give users a notice when they input dismatched city names. 3.There are in total 1179 users who have contributed to the San Jose. Combined Top 10 users contributed 64.19%. So there are many users only contribute a little piece of information.It is neccessary to study the dynamics of the users and find ways to increase their activites. Developing useful applications through OpenStreet map would be a good way to advocate its influence, and grow the community of contributors and users. For instance, like wikipedia that everyone cannot live without, we can also make OpenStreetMap a map version of wikipedia. One thing I could think of is to use Some interesting tags such as "like","recommend" for restaurants, sceneries, entertainment,hiking trails that people cares about in their daily life.

Benefits and problems with additional improvements discussed¶

1.Enhance format standardization: benefits: saving huge amount of time in terms of data munging since information is more aligned in format; for users who creating map data, it's also easier for them to edit the info since there is a gold standard to follow and auto-correction exist to aid the process problems: may not everyone could get used to the enforced format, it's hard to find a gold standard for all kinds of info and for every one to follow due to regional habits & preference. Also, since OpenStreetMap is a global platform, it means a lot of workload to get it done considering the massive number of countries & languages.

2.Autofilter for mistaken information: benefits: avoiding some obvious mistake to enhance the data quality. problems: need to preset the geoinfo association with zipcode and city name, and constantly update the changes. This requires a lot of workload, and may not applicable for every country.

3.User contribution: benefits:more users involved will help to develop a better community, thus the map data will be not only more complete but also more useful & powerful.
problems:current method of collecting and inputing data is not that easy so everyone could execute. Need time and energy to develop better interface and method for users to follow. Also, it's quite capital intensive to extend the functions for a non-profit website which is currently rely on volunteers. There will be a huge challenge to fund this initiative.
